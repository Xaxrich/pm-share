1 · 业务背景

先看一个真实发生的错误

场景：保洁师傅迟到了

用
「怎么还没来，都超了半小时了」
AI
系统识别：服务人员进度查询 ✓（语义正确）
AI
「您好，您的订单师傅正在赶往您的位置，通常会按预约时间上门，请耐心等待……」
用
「我已经等了半小时了你给我说耐心等待！？」

发生了什么

  • 语义上：识别「服务人员进度查询」没有错
  • 业务上：当前已超预约 30 分钟，这是迟到风险，绝不能按普通查询回复
  • 系统把「已超预约」这个业务事实完全忽视了

这不是「AI 聊天质量」问题，这是业务控制问题。语义分类准确，业务动作是错的。

这就是为什么最难的地方不在 NLP，而在于：知道什么时候自动化已经不能继续了。
开场共鸣：语义对了，业务动作错了——后面所有「路由 / 分层 / skills」都从这里长出来。
1 · 业务背景

先别急着讲模型，先看这套系统在替谁做判断

三方角色

用户

要的是快、准、少废话。最怕重复解释、最怕一直被安抚、最怕问题没往前走。

服务方

门店 / 阿姨 / 师傅要的是少打扰、少误伤、少被平台乱升级。

平台

要的是自动化承接更多、少无效转人工、少投诉、还能保住履约结果。

业务目标

  • 效率：能查的自己查到，能闭环的别堆给人工
  • 体验：少让用户重复说，少让用户觉得在绕圈
  • 结果：别把问题解释漂亮了，结果还是没处理

所以这套系统从第一天起就不是“回答系统”，它更像一个处在交易链路里的控制层。

这里要讲得像在上班：谁在乎什么、谁怕什么，不是抽象概念。

1 · 业务背景

「怎么还没来」——同一句话，在不同阶段是完全不同的问题

决定系统怎么处理的，不是句子本身，而是句子所在的业务阶段。

售前
时效咨询
用户问「今天还有档期吗」→ 查服务时效，不查订单
待履约
普通查进度
「师傅大概几点到？」→ 查订单状态，普通回复即可
临近履约 ⚠
迟到风险
已超预约时间 → 不能普通回复，应催单 / 升级 / 给方案
服务中 ⚡
履约异常
出发了却没到？→ 立刻介入，不能再安抚
售后
承诺未兑现
「上次说好的返工为什么没人来」→ 查返工单，是历史问题

意图识别不能只看「句子说了什么」——必须先知道现在处在哪个阶段、已经发生了什么。

本地生活最难的点之一：同一套系统要跑售前 / 待履约 / 服务中 / 售后，同一句话的含义和后续动作可以完全不同。

🎯

2 · 意图识别

一开始大家都会把它理解成分类问题，做着做着就会越来越重

旧思路

一句话进来 → 分到一个标签 → 去场景回复
  • 短期看很顺，尤其在高频场景里
  • 但越往后越会把阶段、对象、风险、流程状态全塞进“意图识别”里
  • 分类器越来越像一个替系统背锅的大黑盒

问题出在哪

真正难的不是给句子找名字，而是系统得先知道：

  • 这句话落在订单生命周期的哪一段
  • 这句话后面系统要推什么动作
  • 这一把如果走错，代价到底大不大

所以意图识别做到后面，一定会从“分类题”变成“路由题”。这不是理论升级，是被业务逼出来的升级。

这里不要讲得太学术。重点是“为什么旧系统越做越重”。
2 · 意图识别

因此更值得先问三个问题，而不是先问“这是什么标签”

Question 1
它出现在订单生命周期的哪个阶段？

因为阶段一变，同一句话后面允许的动作空间就变了。

Question 2
这句话后面最主要要推进什么动作？

查状态、问规则、求执行、求升级，先把大方向定住。

Question 3
这把如果处理错了，业务风险有多大？

风险高的时候，系统不该硬猜，更不该继续普通自动化。

Then
再去收细主题

到这一步再谈订单进度、退款进度、返工、物损，系统会稳很多。

Result
意图识别开始像路由，不再像贴标签

到这里，系统才算从“会分”走向“会处理”。

本节对「意图识别如何定义」的核心表达；讲清即可，不必堆术语。
2 · 意图识别

落到系统里，通常拆成三层，再配三种技术

第一层：生命周期

售前 / 待履约 / 临近履约 / 服务中 / 售后

  • 优先靠世界状态判断
  • 状态不全时再让小模型兜底

第二层：动作层

查状态 / 问规则 / 求执行 / 求升级 / 投诉风险

  • 小模型主判，讲究稳定、低时延、可校准
  • 这一层是整个系统的粗路由骨架

第三层：细主题

订单进度 / 服务人员信息 / 退款进度 / 返工 / 物损等

  • 按阶段拆空间，不做一个总分类器
  • 低置信度、边界类再交给 LLM 复判

也因此不必过度纠结“到底选哪一个模型”。先把层次理顺，模型选择反而会变简单。

这页把“分类/技术方案/业务定义”三块合在一起，不要再拆太散。

③ 动作层
用户推动方向 → 求执行 / 求升级，不是普通「查状态」
④ 控制层
归一

自动锁定：今日唯一活跃订单 ← 不必先要订单号
⑥ 风险门控
STOP 自动化 → 允许：催单 / 建单 / 转人工；禁止：继续「请耐心等待」式安抚
⑦ 执行层
触发催单升级 Skill：承认超时 + 已联系门店 + 15 分钟内反馈

Channel Strategy

不同渠道，为什么一定要分开设计

这个地方最容易被做浅。不是 IM 是文字、400 是语音这么简单，真正不同的是系统责任。

A输入质量不同
B用户对时延和追问的容忍度不同
C每个渠道想优化的结果也不一样
第三章重点不是技术适配，而是系统责任差异。
3 · 渠道策略

同一套意图体系可以共用，但识别策略不能照搬

输入质量

IM 文本更完整，企微上下文更连续，400 还有 ASR 噪声。系统面对的不是同一种原材料。

时延容忍

IM 可以多轮澄清，企微可以持续推进，400 场景通常容不得系统连问多轮。

系统目标

IM 更适合自动化承接，企微更适合推进处理，400 更适合风险分流。

所以这里不是“一个模型上三个渠道”，而是“一个语义骨架，三套控制策略”。

这页是总判断。下一页再把 IM / 企微 / 400 展开。
3 · 渠道策略

三种渠道可以这样理解：不是三个入口，是三种系统责任

IM

自动化承接主战场

  • 允许细一点，允许问清楚，允许多走一步 skill
  • 重点是把用户问题尽量导进可执行链路

企微

状态推进主战场

  • 用户很多时候不是来提新问题，是来追上次那件事
  • 重点是识别当前推进到哪一步、接下来怎么往前推

400

风险分流主战场

  • 第一优先级不是分得多细，是别把高风险问题继续压在自动化里
  • 这里宁可保守一点，也别硬撑
这一页收束：不同渠道策略不同的底层原因。
3 · 渠道策略

三个渠道的「失败」长得不一样——所以策略不能抄

IM

PM 设计重点：自动化路由尽量深

失败 ≈ 该自动闭环的没吃到，问题白白堆给人工

企微

PM 设计重点：会话状态机 + 推进行为

失败 ≈ 推进节奏断掉，用户反复从头描述

400

PM 设计重点：风险分流，宁可保守

失败 ≈ 该退出自动化时没退出，矛盾被激化

同一套语义骨架可以共用，但识别策略、澄清轮次、升级门槛要按渠道单独调——否则是在用 IM 的成功标准去坑 400。

从「责任」落到「各自怎么失败」，渠道章更有记忆点。

4 · skills 落地

为什么系统最后会从决策树和大 prompt 走向 skills

旧做法做到后面会遇到什么

  • 规则越来越多，状态越来越多，prompt 越来越长
  • 经验散在不同人脑子里，新人接手很慢
  • 线上出问题时，很难说清到底是识别错、路由错还是执行错

skills 带来的变化

  • 经验开始被固化成能力模块
  • 系统不再把所有逻辑都压给总 prompt
  • 每个模块终于可以讲清职责边界：负责什么、不负责什么

所以 skills 真正带来的不是“更高级”，而是“系统终于能把经验模块化、边界化、可评估化”。

这一页要讲得很实。skills 不是为了追新，是因为旧做法已经承载不住复杂性。
4 · skills 落地

结合业务里在用的 openClaw 底座，skill 其实是在做这四件事

Step 1
扫描

扫描内置和项目目录下的 SKILL.md，把能力清单先暴露出来。

Step 2
识别

根据 name / description / title 等元信息，判断哪些 skill 值得进一步读。

Step 3
读取

运行时再去读具体 skill 文档，而不是一开始把所有经验都塞进上下文。

Step 4
执行

基于 skill 文档里的约束和步骤，走 reAct + tools，把事办掉。

Value
渐进式披露

上下文更省，经验更模块化，系统也更容易维护。

因此更合理的判断是：skills 不是“prompt 换壳”。其真正价值，在于把经验的暴露方式从“一次性全塞进上下文”改成“按需读取”。

与项目中的 lbgClaw / openClaw 机制对齐即可，不必展开底层代码。
4 · skills 落地

一个成熟的 skill，至少要把这几件事写清楚

标准组成

  • name：这个能力到底叫什么
  • description：什么场景应该触发它
  • title：和旧系统、旧场景怎么映射
  • 执行步骤：先做什么，再做什么，失败了怎么办
  • 输入契约：需要什么对象、缺了怎么补
  • 输出契约：回复、push、转人工、建单分别怎么处理

拿订单查询 skill 举例

  • 先看有没有订单对象
  • 没有就走对象解析：自动补、候选确认、最后再追问
  • 有对象就查订单详情
  • 根据当前时间、履约状态和用户关注点决定回复还是升级

可以看到：好的 skill 不该从零猜问题，而应接住已被前置层收窄过的问题，再把它闭环掉。

这里用真实 skill 例子最有说服力。讲“订单查询 skill”比讲抽象 skill 更有效。
4 · skills 落地

skills 最容易踩的坑，建议在团队里重点盯这四个

误触发

description 写得太泛，或者前置层没收窄好，skill 被错误问题触发。

边界冲突

两个 skill 都觉得自己能接，最后靠 prompt 抢触发，系统会越来越乱。

职责混装

skill 里面又做意图判断、又做对象层、又做风控，最后重新长成一个大黑盒。

只做不评

不知道它误触发率多少、fallback 多少、执行成功率多少，skill 就会失控。

所以从单个 skill 走到完整能力体系，最后一定要有四样东西一起协同：tool、skill、policy、eval。

这一页就是 skills 章节的收束页：不是只会写 skill，还要知道 skill 会怎么坏。
贯穿 · 真实教训

三个「差一点就这么做」的坑

分别对应：状态 / 对象 / 风险门控——和前面管道、trace 一一可对上。

Bad Case 01

语义对了，业务动作错了

「怎么还没来」识别成服务人员进度查询——语义准；但已超预约 30 分钟仍回复「请耐心等待」→ 用户崩溃升级。
💡

评价维度应是业务动作对不对，不是只看分类 F1。超预约是事实，不是语义难题。

Bad Case 02

对象拿错了，Skill 白跑

用户问今天保洁进度，系统从文本抽到 ID 碎片却关联到昨天已完成订单——回复「正确」但完全答非所问。
💡

拿错对象比缺字段更危险；对象解析必须是系统级统一能力。

Bad Case 03

没有刹车，系统不该继续却继续

连续三次不满仍「安抚 + 自动回复」，第三次后用户威胁投诉——若有 risk gate，第二次就应停自动化转升级。
💡

优先级解决排序，解决不了「何时必须停」；风险门控要独立成层。

可与 trace、开场对话对照讲，收束第 2～4 章的共同主线。

PM Value

最后讲最现实的一件事

系统讲完了，最后一定会回到一个问题：这件事里，AI PM 的不可替代价值到底在哪。

A不必替代算法在其强项上的工作
B把问题定义、标签体系、评估口径与业务结果的主导权落在产品侧
C建设节奏落在可交付里程碑上，避免停在大蓝图
最后一章宜真诚、可落地，避免空泛总结。
5 · PM 价值

不是调模型，是定系统的边界和结构

容易误解成在做的事

🤖只追「最强模型」
📝把 Prompt 写得更长更全
🎯单靠抬意图分类 F1 救系统
📊只盯「AI 解决率」一个数

更值得投入的事

🗺系统应该知道哪些业务事实（世界状态）
📐原生诉求 vs 流程状态 vs 风险信号的语义坐标系
🛤生命周期骨架 + 动作路由骨架
⚖️何时可继续自动化、何时必须停
📋每个 Skill 的执行契约与评估口径

典型产出物：

Taxonomy 主表Boundary BookRisk Ontology Object PolicySkill 契约Eval 口径
上班感：外行也能一眼看到 AI PM 的交付物长什么样。
5 · PM 价值

一个越来越清晰的判断：AI PM 更该建立主导权的领域，不在模型结构里

不必与算法在这些环节角力

  • 具体模型结构怎么搭
  • 训练细节怎么调
  • 参数怎么设更优
  • 某个单点 benchmark 谁高谁低

这些环节算法侧天然更强；即便压过一头，对业务收益也有限。

更值得建立主导权的是这些

  • 问题到底怎么定义
  • taxonomy 到底怎么分层
  • 系统看什么指标才算真的有效
  • 自动化边界画在哪里
  • 最后到底对什么业务结果负责

这些东西不先定义清楚，算法优化越努力，系统可能越偏。

因此更务实的结论是：AI PM 最值钱之处，不在替算法想模型，而在为系统把“什么叫对、什么叫该停、什么叫真的解决”界定清楚。

这一页宜结合亲历案例，增强可信度。
5 · PM 价值

现实里最常见的几类摩擦，实务里往往这样对齐

算法
“我们先把分类器做准，后面自然会好很多。”
→
PM
“先别急。taxonomy 还没拆清，分类器越准，可能只是在一个混乱体系上提分。”
工程
“我先把几个 skill 接起来，把链路跑通。”
→
PM
“先把对象层和 risk gate 立住。不然 skill 会越来越胖，后面全得重来。”
业务
“AI 解决率能不能再提一提？”
→
PM
“先把‘什么叫真的解决’定义清楚。不转人工不建单，不等于用户问题已经被解决了。”
这页突出「上班感」：贴近一线协作与摩擦，避免纸上谈兵。
5 · PM 价值

最后：建设路径可切成回合，避免永远摊在大蓝图上

1
第一回合 · 底盘

先把世界和语义骨架立住

世界状态、生命周期、动作层、taxonomy。先让系统知道自己现在在哪。
2
第二回合 · 收窄

再把收窄和刹车补上

Query 归一、对象层、risk gate。高风险与高不确定性先控住。
3
第三回合 · 能力

最后再重构 skill，做学习闭环

skill / tool / policy / eval 协同，OOS 驱动系统长出新能力。

进入下一回合的前提：当前回合最核心的错误，已从「结构性错误」降级成「局部边界问题」——不是模块上线了就能推进。

这一页是整场的落地收束。讲顺序比讲模块更重要。
收官

最后想留住的，是一种看 AI 产品的方式

大模型不会自动长成业务系统。 模型侧做生成与改写，系统侧对结局、合规和客户体验负责。产品要把生成能力收进一张能追责、能核对的图里。

看世界

先把订单、阶段、对象、历史承诺这些业务事实摆清楚，再听用户具体说了什么。许多卡住的「识别」，根子在世界状态没补齐，不在模型少认了几个字。

定边界

费工夫的是写清哪些交给状态与规则、哪些交给模型推断、哪些必须走工具和流程。这张图糊了，后面只会越做越乱。

知道何时停

一线返听里，更麻烦的多半是该停下来的地方没有停。成熟度的分水岭在边界与风险可核对、可复盘，话术像不像人可以往后排。

真正值得带回现场复盘的一个问题，是当前这套系统里，哪一层最常在替别的层扛结局。

  • 业务事实是否仍在让模型猜
  • 风险边界是否仍挂在意图优先级上
  • Skill 是否已经胖到在吃前置层的不确定性
  • 「其他意图」究竟是垃圾桶，还是新能力入口
